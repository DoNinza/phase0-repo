## 원본과 다른 점 (의도적 변경, 수치 아님)

- `t_annual()`은 이제 하드코딩된 4개 시나리오만 받지만, 원본과 완전히 동일한
  출력을 낸다 — 구조만 재편.
- **미세 발견 1건**: 원본 `t_cost_sens`는 Conservative 비용을 `.0047`로
  하드코딩했는데, 이는 `Base(0.36%) × 1.3 = 0.468%`를 표시 단계에서 반올림한
  값(0.47%)을 계산에 재사용한 것이다. 반올림 없이 그대로 계산하면 `E_trade =
  -0.048%`이며, 원본 표기(`-0.050%`)와 0.002%p 차이가 난다. 판정 방향에는
  영향 없음(둘 다 음수 → Conservative 미통과는 동일)이지만, 이 자체가
  기획안이 R-12에서 지적한 "표시값을 다시 계산에 흘려 넣는" 패턴의 축소판이라
  일부러 남겨두고 테스트로 고정했다(`tests/test_tables.py`).

## Phase 0 Blocker 해소 (§5.6) — 2026-07-14

`costs.yaml`의 `외부 확인 필요` 항목 2건이 모두 실측값으로 대체되어 Blocker가
해소됐다. Base 왕복비용은 **0.36%(문서 원본 가정치) → 0.3854%(실측)** 로 갱신:

- `brokerage_commission_roundtrip` — 사용자 실계좌(한국투자증권 뱅키스 온라인매매)
  확인 결과 0.0140527%(편도, KRX). 왕복 2배 적용 = 0.0281054%. 기존 0.01%
  임시값 대비 약 2.8배 상향.
- `exchange_fees` — KIS 공식 수수료안내 페이지 기준 유관기관제비용율
  0.00363960%(편도, 2025-10-27 고시). 왕복 추정 0.0073%로 Base에 반영.

이 갱신에 따라 `t_r01_verify`처럼 하드코딩된 예시(원표 조건 C=0.3% 고정)를
제외한 모든 표·테스트가 새 Base(0.3854%) 기준으로 자동 재계산되며,
`tests/test_costs.py`·`tests/test_tables.py`의 관련 assertion도 이 커밋에서
함께 갱신했다. 앞으로 비용이 또 바뀌면 `costs.yaml`의 `value`만 고치고
`pytest`를 돌려 실패하는 assertion만 새 값으로 맞추면 된다.

## Project Blocker 해소 (§5.6) — 2026-07-14

KIS 약관상 자동매매(Open API 기반 개인 알고리즘 거래) 금지 조항 여부를 사용자가
직접 확인 — **금지 조항 없음**. 조사 과정에서 함께 확인된 정황: KRX 알고리즘거래
사전신고 의무는 개인이 아닌 회원(증권사) 대상이며 파생상품계좌 한정 조항이라
이 프로젝트(KOSPI 주식 데이트레이딩)와는 무관, "프로그램매매" 규제(사이드카 등)는
동시 5억원 이상 대량 주문의 기관·외국인 대상이라 개인 알고리즘 거래는 해당 없음,
업계 전반(KIS 2022~, 토스증권 2026~)이 개인용 오픈API 자동매매를 활발히
마케팅하는 정착된 상품 카테고리. 단, 이용약관 원문 전체를 읽고 확정한 법률
자문은 아니며 사용자 본인 확인에 근거한다.

## pykrx 실데이터 연결 (STAGE 7 항목 6·7) — 2026-07-14, 부분 완료

`phase0/data/pykrx_ingest.py`로 실제 KRX 일봉 데이터 수집·클렌징 파이프라인을
연결했다. `scripts/ingest_real_sample.py`로 실제 네트워크 호출을 확인함:

- **개별 종목 시계열(`get_market_ohlcv`)은 정상 동작** — 삼성전자(005930)
  2026-07-01~14 10봉을 실제로 수집·클렌징 성공.
- **전체 시장 스냅샷(`get_market_ticker_list`, `get_market_cap`)은 현재
  깨져 있음** — KRX 서버 응답 형식이 바뀌어 `Expecting value: line 1 column 1`
  JSON 파싱 오류가 남. pykrx 저장소에 동일 증상의 미해결 이슈가 여러 건
  존재한다(sharebook-kr/pykrx#190, #191, #193) — 이 프로젝트 코드의 결함이
  아니라 pykrx·KRX 쪽 상위 장애다.
- **실질적 영향**: 항목 6(10년 일봉·시가총액·거래대금 수집)은 종목을 이미
  아는 경우 가능하지만, 항목 7(상장폐지 종목 포함 여부, 생존편향 방지)에
  필요한 "과거 시점 전체 유니버스 목록"은 이 상위 장애가 풀리기 전까지 이
  경로만으로는 완결할 수 없다. `fetch_universe()`는 이 실패를 조용히 삼키지
  않고 `UniverseUnavailableError`로 명시적으로 알린다.
- 클렌징 로직(휴장일 0봉 제거, OHLC 정합성 검증)과 유니버스 실패 처리는
  fetcher를 주입받는 구조라 네트워크 없이 pytest로 검증된다
  (`tests/test_pykrx_ingest.py`, 9건).

**다음 단계**: pykrx 업스트림 수정을 기다리거나, KRX 정보데이터시스템 등
대체 소스로 과거 유니버스(상장폐지 포함)를 구해야 항목 7이 완결된다.

## 후보 종목 배치 수집 (STAGE 7 항목 4, v3 T-03) — 2026-07-14

`phase0/data/candidate_batch.py`: 후보 리스트 → 종목별 일봉 수집 → 유동성
요약(평균 일 거래대금 프록시 = mean(종가×거래량))까지 배치로 처리한다.
`scripts/run_candidate_batch.py`로 실제 KRX 데이터 5종목에 대해 실행 확인
(2026-06-01~07-14, 5/5 성공, 유동성 필터링까지 정상 동작).

- 종목 단위로 실패를 격리 — 한 종목이 상장폐지·네트워크 오류로 실패해도
  나머지 배치는 계속 진행하고, 실패 사유를 `TickerSummary.error`에 남긴다.
- `DEFAULT_CANDIDATES`는 **손으로 고른 대형·고유동성 KOSPI 종목 20개
  예시(플레이스홀더)**다 — 항목 7(전체 유니버스 스냅샷)이 막혀 있는 상태에서
  파이프라인 자체를 실데이터로 검증하기 위한 목록일 뿐, 실전 후보 선정
  근거가 아니다. 실제 후보 50종목은 항목 7이 풀리거나 사용자가 직접
  확정해야 한다.
- 테스트(`tests/test_candidate_batch.py`, 5건)는 fetcher를 주입해 네트워크
  없이 배치 격리·필터링 로직만 검증한다.

## 분봉 이력 범위 (STAGE 7 항목 5, 분봉 검증 Blocker) — 2026-07-14, 잠정 조사 완료

계좌 접근 없이 공개 API 문서·커뮤니티 자료로 조사한 결과, KIS 표준 분봉
조회 API(`FHKST03010200`, "주식현재가 분봉조회")는 **당일(오늘) 데이터만
제공**하는 것으로 보인다 — 1분봉 단위, 1회 호출당 30건(30분) 한정, 과거
날짜의 분봉은 이 API로 조회 불가(일봉·주봉·월봉은 과거 조회 가능한 것과
대비됨). 즉 "이력 범위가 며칠인가"가 아니라 "이력 자체가 없음(당일 한정)"이
잠정 결론이다.

**주의**: 공식 API 스펙 원문(PDF/명세 페이지)을 직접 읽어 확정한 것이
아니라 커뮤니티 자료 기반이며, 확정하려면 실제 계좌 APP Key/Secret으로
직접 호출해보는 검증이 필요하다 — 사용자가 본인 계좌로 직접 확인 예정.

**설계에 미치는 영향(아직 코드에 반영 안 함)**: `g0_backtester.py`는 G0
"부호 갈림"(낙관>0, 보수≤0) 케이스를 "분봉 표본으로 재판정 1회"로 풀도록
설계돼 있는데, 위 조사가 맞다면 그 재판정은 과거 데이터로 소급할 수 없고
해당 상황이 다시 발생할 때 실시간으로만 관찰 가능하다 — "과거를 캐서
검증"이 아니라 "앞으로 관찰해서 검증"으로 바뀌어야 할 수 있다. 사용자 본인
확인 후 필요하면 반영한다.

## 남은 것 (STAGE 7 B·C그룹 — 계좌·API 접근 필요)
